# 1.可以先把词表的向量先计算出来保存吗？
完全可以，而且强烈推荐。
将前置词和异常词的向量预计算并持久化，可以避免在每次增强时重复调用模型，大幅提升运行效率。

推荐做法
词向量模型选择：使用轻量级、覆盖中文良好的模型，例如：

Tencent AI Lab Embeddings（200 维，约 100 万词，内存友好）

fasttext-wiki-news-subwords-300（Facebook，支持 OOV）

sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2（384 维，语义丰富，但需 GPU）

保存格式：

字典形式：{word: vector}，保存为 pickle、json（需转 list）或 h5py / numpy 的 .npy + 词汇表索引。

更高效：使用 faiss 索引，支持快速近似最近邻搜索。

预处理：

对词表中的所有 prev_word 和 abnormal_word 去重后计算向量。

# 2.不同的模型需要分别计算吗?
取决于下游增强时使用的「匹配策略」是否依赖同一个向量空间。

情景 A：只用于语义相似度筛选（例如找与正确词最像的异常词）
若 始终使用同一个词向量模型（比如你固定选用 paraphrase-multilingual-MiniLM），则只需计算一次，所有增强任务共用。

若后续更换词向量模型（例如从静态词向量换到 BERT 动态嵌入），则需要重新计算。

情景 B：增强时使用了不同模型（如 GPT 生成噪声句子）
若你只是借助词向量做候选筛选，筛选后仍然做简单的字符串替换，则原向量足够，不需要为 GPT 重新计算。

若你打算用 GPT 的嵌入空间（例如获取 GPT 对词的表示）来计算相似度，则需要用 GPT 的 tokenizer 和 hidden state 重新抽取向量（成本较高）。

建议
采用独立的、轻量级的词向量模型（如 paraphrase-multilingual）专门做相似度匹配，与对话增强的主模型（如训练用的 LLM）解耦。

这样词向量只需计算一次，可长期复用。


# 3.可以使用什么模型用来计算匹配比较好？
匹配需求：给定一个正确词（出现在对话中），从候选异常词中找出最“自然”的 ASR 错误替换词。
ASR 错误通常包含三种模式：发音相似、字形相似、语义相近。建议融合多种相似度，效果更佳。

推荐组合方案
相似度类型	推荐模型 / 方法	权重
语义相似	paraphrase-multilingual-MiniLM-L12-v2	0.4
拼音相似	pypinyin 转换为拼音后计算编辑距离（归一化）	0.4
字形相似	五笔编码（python-xwubi）或笔画编辑距离	0.2
为什么不只用语义？
因为 ASR 错误往往产生发音相近但意思完全不同的词（例如「还款」→「还宽」），仅靠语义相似度可能选不到合适的异常词。
